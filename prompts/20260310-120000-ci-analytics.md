## Human

Implement CI analytics for robotocore -- track test reliability and service usage across CI builds. Requirements include CI session tracking, per-session analytics, cross-session analysis, reliability tracking, and management endpoints.

## Assistant

## Key decisions

**Architecture**: CI analytics is implemented as a new module `src/robotocore/audit/ci_analytics.py` that follows the same patterns as the existing audit log (singleton, thread-safe, lazy initialization). The module auto-detects CI environments from standard env vars (GITHUB_ACTIONS, GITLAB_CI, JENKINS_URL, CIRCLECI, CI=true) and activates only when CI is detected or explicitly enabled.

**Session tracking**: Each CI run gets a `CISession` object that tracks request counts per-service, operation success/failure rates, timing, and services used. Sessions are identified by `ROBOTOCORE_CI_SESSION` env var or auto-generated UUID.

**Persistence**: Session summaries are saved as JSON files in `ROBOTOCORE_STATE_DIR/ci_analytics/`. This allows cross-session analysis without requiring a database.

**Integration**: CI analytics hooks into the existing `audit_response_handler` in the handler chain, so every AWS request is automatically tracked when CI mode is active. Session is auto-saved on server shutdown.

**Endpoints**: Four management endpoints under `/_robotocore/ci/` for listing sessions, getting details, computing aggregate summaries, and clearing history.

**Test-first approach**: 34 unit tests written before implementation covering CI detection, session lifecycle, persistence, aggregation, and reliability metrics.
