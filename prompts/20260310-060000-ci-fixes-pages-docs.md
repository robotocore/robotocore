### Key decisions

**Cognito dual-store problem**: Operations in `_ACTION_MAP` write to `CognitoStore`;
ops not in the map fall through to Moto which has no knowledge of natively-created
pools/users. The fix was Moto-sync helpers (`_sync_pool_to_moto`, `_sync_user_to_moto`,
`_sync_client_to_moto`) called after each create op, plus implementing 7 token-based
operations natively (`GlobalSignOut`, `ConfirmSignUp`, `UpdateUserAttributes`, etc.)
using a JWT-decode helper that extracts `sub` from the access token to locate the user
by UUID rather than username. The alternative — making all ops go through Moto — would
have required removing our richer JWT generation. This keeps both stores consistent.

**CI cache removal**: `astral-sh/setup-uv@v4` renamed the parameter from `cache: true`
to `enable-cache: true`, but GitHub's cache service was also returning intermittent 400
errors causing job failures. Tried the rename first; still failing. Removed caching
entirely from all workflows. The speed hit is acceptable for correctness.

**App test date window**: Two `get_metric_statistics` calls had hardcoded
`StartTime/EndTime` that became stale overnight. Replaced with `datetime.now(UTC) ± 1h`
so they stay green indefinitely.

**Docker SQS smoke test**: `receive_message` returned empty 10× even after
`send_message` succeeded. Root cause: system boto3 was 1.34.46 (vintage) vs our
server's SQS implementation. Fixed by upgrading boto3 in the smoke test runner,
adding 0.5s sleep, and increasing retries to 10 with `MaxNumberOfMessages=10`.

**Banner SVG baseline**: The tagline "a digital twin of AWS" was at `y="140"` with
`font-size=15`. Text baselines sit at the declared y, but glyphs extend ~12px upward.
The decorative horizontal bar was at `y="132"`, cutting through the middle of descenders.
Moved tagline to `y="152"` and the badges group down proportionally.

**Coverage SVG design**: Chose an animated 21×7 dot grid (one dot per service) over a
bar chart or heatmap — more visually interesting, scales cleanly to 147 services, and
the animation (wave reveal, staggered by column then row) makes it feel alive. Native
providers get an outer ring. Color tier mapping: gold (90-100%), amber (70-89%),
olive (40-69%), muted green (1-39%), pale (0%).

**GitHub Pages architecture**: Single-file HTML with `DATA_PLACEHOLDER` literal replaced
by `json.dumps(data)` at build time (via `scripts/build_pages.py`). Chose this over
fetching JSON at runtime to avoid CORS/network issues and make the page work offline.
The build script runs `generate_parity_report.py` then injects the result. Workflow
uses `actions/upload-pages-artifact@v3` + `actions/deploy-pages@v4` with
`build_type=workflow` (configured via `gh api`).

**Interactive docs redesign**: The original page had 6px service names — unreadable.
Rebuilt as a proper documentation page: sticky header, searchable table with 14px+
fonts, expandable rows. Each expanded row shows a boto3 connection snippet
(pre-filled with `endpoint_url='http://localhost:4566'`) and the full operation list
color-coded by implementation source (native/moto/missing) with required parameters
extracted from botocore shapes. Chose a table-primary layout over a sidebar because
147 services fits well in a flat list with good filter+sort controls.

**Op params in parity report**: Extended `generate_parity_report.py` with
`get_operation_params()` (reads botocore `required` fields per op) and added `all_ops`

(~2-3MB) for a static site loaded once.
