## Key decisions

**Architecture**: Created `src/robotocore/services/apigatewayv2/websocket.py` as the WebSocket transport layer. It handles the full ASGI WebSocket lifecycle (accept, receive loop, close) and bridges to the existing executor functions (`execute_websocket_connect`, `execute_websocket_message`, `execute_websocket_disconnect`).

**Routing convention**: WebSocket connections use path `/ws-exec/{api_id}/{stage}`, mirroring the existing `/v2-exec/{api_id}/{stage}` convention for HTTP APIs. The `_resolve_ws_api()` function matches this pattern.

**Server-to-client push via queues**: Each live WebSocket connection gets an `asyncio.Queue` stored in `_send_queues` keyed by `(api_id, connection_id)`. A dedicated `_send_loop` coroutine drains the queue and pushes messages through the ASGI send callable. The `push_to_connection()` async function and the existing `post_to_connection()` sync function both write to this queue.

**Chose asyncio.Queue over callbacks**: Queues decouple the producer (Lambda integration calling @connections POST) from the consumer (ASGI send loop). This avoids holding references to raw ASGI send callables in the provider layer, which would create a tight coupling between the transport and the data layer.

**Enhanced existing post_to_connection**: Rather than replacing the existing sync `post_to_connection()` in the provider, added a non-blocking `queue.put_nowait()` call alongside the existing message storage. This preserves backward compatibility -- the @connections POST API continues to work whether or not a live WebSocket is connected.

**ASGI middleware integration**: Added WebSocket scope handling as the first check in `AWSRoutingMiddleware.__call__`. WebSocket scope is dispatched immediately without going through the HTTP handler chain (no auth/chaos/audit needed for the transport layer itself -- the Lambda integrations handle their own auth).
