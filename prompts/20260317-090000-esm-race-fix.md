## Key decisions

**Root cause**: The CloudFormation event pipeline template has an
`EventSourceMapping` with `Enabled: true` that polls the SQS queue every
1 second (`time.sleep(1)` in `event_source.py`). The test calls
`send_message` then immediately `receive_message`, but the ESM poller
races for the message and sometimes consumes it first, leaving the test's
`receive_message` with an empty queue.

The first two tests (`test_sqs_message_roundtrip`, `test_dynamodb_item_roundtrip`)
often pass because they run early before the poller has had a chance to
establish its rhythm. The third test runs later, by which point the poller
is reliably firing on a 1-second cadence.

**Fix**: Add an `EsmEnabled` parameter (default `"true"`) to the template
with a `Conditions` block. `test_sqs_multiple_messages` deploys with
`EsmEnabled=false`, which sets `Enabled: false` on the EventSourceMapping
via `!If [EsmEnabledCondition, true, false]`. The poller never starts for
this stack, so the test owns the queue exclusively.

**Why not retry in the test**: If the ESM consumed the message, it's gone
from the queue and can't be retried. The fix must prevent the race, not
recover from it.
