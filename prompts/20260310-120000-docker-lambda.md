## Key decisions

**Executor pattern**: Created `DockerLambdaExecutor` as a self-contained class that mirrors the existing `RuntimeExecutor` protocol but adds a `runtime` parameter to select the Docker image. This avoids modifying the existing runtime registry — Docker execution is an orthogonal axis (where to run) vs (which language).

**Dispatch strategy**: Modified `invoke.py` to check `LAMBDA_RUNTIME_EXECUTOR` env var at invocation time, dispatching to either the Docker executor singleton or the existing per-language executor. This keeps the two paths completely independent — no risk of breaking the existing local execution.

**Fallback design**: If Docker SDK is missing or the daemon is unreachable, `DockerLambdaExecutor` silently falls back to `execute_python_handler()` with a warning log. This means setting `LAMBDA_RUNTIME_EXECUTOR=docker` in an environment without Docker doesn't break anything.

**Warm pool**: Implemented `WarmContainerPool` with LRU-style per-function container tracking, configurable keepalive via `LAMBDA_KEEPALIVE_MS`, and optional container removal. Pool uses a threading lock for safety since Lambda invocations run in a thread pool.

**Image mapping**: Hardcoded defaults for all major AWS Lambda runtimes (python, nodejs, java, ruby, dotnet, provided). Custom overrides via `LAMBDA_RUNTIME_IMAGE_MAPPING` env var accepting either JSON string or file path — same pattern LocalStack uses.

**What I skipped**: Actual Runtime Interface Emulator (RIE) integration — the current implementation passes the event as the container command and reads stdout/stderr. A future enhancement would implement the full Lambda Runtime API (`/2018-06-01/runtime/invocation/next`) via a sidecar HTTP server, which is what AWS Lambda base images expect. The current approach works for simple handlers but won't work with unmodified Lambda base images that expect the RIE protocol. This is a known gap to address in a follow-up.
