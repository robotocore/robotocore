## Key decisions

**Architecture**: Cloud Pods is a transport layer on top of the existing
StateManager. It uses `StateManager.save()` to create the state files, then
compresses them into a tar.gz archive and uploads to the backend. Loading
reverses the process. This avoids duplicating any serialization logic.

**Versioning**: Each save creates a new version with microsecond-precision
timestamps (`v20260310T120000.123456`) to avoid collisions on rapid saves.
Versions are stored as `{backend}/{name}/v{timestamp}.tar.gz`.

**Backend abstraction**: The CloudPodsManager handles both local filesystem
and S3-compatible backends. S3 backend uses boto3 directly, making it work
with any S3-compatible store (MinIO, another robotocore instance, AWS S3).
Local backend is just directories and files.

**Management API**: Added 5 endpoints under `/_robotocore/pods/` matching
the existing management endpoint pattern in app.py. Uses lazy imports to
avoid circular dependencies.

**Test approach**: 22 unit tests covering all operations (save, load, list,
delete, info, versioning, S3 mock, disabled backend, error cases) plus
6 integration tests covering end-to-end flows and management API endpoints
via Starlette TestClient.
