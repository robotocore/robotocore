## Key decisions

**This prompt log covers Docker-based Lambda execution** (PR #111).

**subprocess over docker-py**: Using `subprocess.run(["docker", "run", ...])` instead
of the `docker` Python SDK. This avoids adding a dependency (docker-py pulls in
requests, urllib3, websocket-client) and the docker CLI is always available where
Docker itself is. The tradeoff is slightly more string wrangling for command
construction.

**Fallback to in-process**: When `LAMBDA_EXECUTOR=docker` but Docker is unavailable
(daemon not running, `docker` not on PATH), the executor falls back to in-process
Python execution and logs a warning. This prevents hard failures in CI environments
that don't have Docker. Non-Python runtimes return an error explaining Docker is
required.

**20 runtime mappings**: All current AWS Lambda runtimes mapped to their official
`public.ecr.aws/lambda/*` images. New runtimes can be added by extending the
`RUNTIME_IMAGES` dict.

**Container lifecycle**: Containers are created with `--rm` for auto-cleanup.
On timeout, `docker rm -f` is called explicitly as a safety net. Lambda environment
variables are passed through via `-e` flags. `AWS_ENDPOINT_URL` is set to
`http://host.docker.internal:{port}` so Lambda functions can call back to
robotocore services.
