## Key decisions

In the previous session I committed work directly to local `main` and tried
`git push`. The remote rejected it because `main` is protected (8 required
checks). I then created a feature branch off the existing commit and pushed
that, but the cleanup step (`git branch -f main origin/main`) had to happen
*after* the failed push — a step easy to forget.

The fix is in CLAUDE.md's "Commit cadence" section because that's where the
old (wrong) guidance lived: *"Every push to main is a release."* That line
encouraged committing on main and pushing to publish. With branch protection
that flow doesn't work; with branch protection it shouldn't work, because the
PR is the unit of review and CI gating.

Three new rules added at the top of the section, before the existing cadence
rules:

1. **Always branch first; never commit on main.** Includes the exact recovery
   sequence if a commit has already landed on local main: move the commit to
   a branch (`git checkout -b feat/<slug>` carries the working commit),
   then `git branch -f main origin/main`. No retry of the push.

2. **Every change ships via a PR.** No direct merges, no
   `git push origin <branch>:main`, no fast-forwarding main locally.

3. **Open the PR as soon as the first commit is pushed.** Use `gh pr create`
   in the same turn, with a real title and body. Iteration on review
   comments is part of the workflow, not a separate phase.

The existing "Push after every commit" line was updated to clarify that the
push goes to the feature branch (which triggers CI on the PR), and that the
CalVer + Docker auto-publish happens after the PR is merged — not on the
direct push that used to be the model.

I also scanned `.claude/skills/*.md` and `.claude/commands/*.md` for any
guidance that mentions main-branch pushes/merges/commits. None did — they're
all task-specific (lint, promptlog, review-pr for inbound). CLAUDE.md is the
right (and only) place for this rule.

Shipping this as a separate, narrowly-scoped PR ahead of resuming work on
PR #275 so the rule is in place before the next implementation session.
