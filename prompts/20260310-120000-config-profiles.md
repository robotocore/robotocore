## Key decisions

**Module location**: Created `src/robotocore/config/` as a new top-level
module rather than putting it in `utils/` because configuration is a
first-class concern that other modules depend on.

**Loading order**: default.env < named profiles (in order) < real env vars.
This follows the principle of least surprise -- more specific overrides
less specific, and explicit env vars always win.

**Variable expansion**: Implemented `${VAR}` expansion that resolves from
already-parsed values in the same file first, then falls back to existing
env vars. This allows profiles to be self-referential.

**Module-level state**: Used module-level `_active_profiles` and
`_resolved_values` to track what was loaded, enabling the management
endpoints to report state without re-parsing files.

**Startup wiring**: `load_config()` is called at the very top of `main()`,
before even reading ROBOTOCORE_HOST/PORT, so profile values can configure
those settings.

**Test-first approach**: Wrote 24 tests covering dotenv parsing, profile
loading priorities, directory creation, and management endpoint data
before implementing any production code.
