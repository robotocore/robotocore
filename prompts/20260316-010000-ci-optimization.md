### Root cause analysis

The 20-minute `compatibility-tests` CI job had three compounding causes:

1. **HTTP 500s from Moto bugs triggering boto3 legacy retry backoff.** boto3
   retries 500 responses up to 4x with exponential backoff, adding ~10s per
   affected test. Three Moto operations crashed with `AttributeError`/`KeyError`
   instead of returning proper 4xx errors:
   - `Athena.GetPreparedStatement`/`DeletePreparedStatement`: `'NoneType' object
     has no attribute 'statement_name'` → should be `ResourceNotFoundException`
   - `OpenSearch.UpdateDomainConfig`: bare `dict[key]` access → should be
     `ResourceNotFoundException` (404)
   - `Workspaces.DescribeTags`: `KeyError` on fake resource ID → AWS returns
     empty tag list, not an error

2. **Single-job parallelism**: All 11,000+ compat tests ran in one job with `-n4`
   workers on a 2-core GitHub Actions runner.

3. **`--reruns=2 --reruns-delay=1`**: The flakiness being retried was *caused by*
   the 500s above (boto3 raising exceptions after exhausted retries). Removing
   reruns eliminates the 3x time multiplier for affected tests.

### Changes

**vendor/moto** (Moto fork, pushed to `JackDanger/moto` master):
- `moto/athena/models.py`: `get_prepared_statement` now raises
  `AthenaClientError("ResourceNotFoundException")` instead of returning `None`
- `moto/opensearch/models.py`: `update_domain_config` uses
  `self.describe_domain()` which raises `ResourceNotFoundException` instead of
  bare dict access
- `moto/workspaces/models.py`: `describe_tags` uses `.get()` and returns `[]`
  for unknown resource IDs (AWS behavior)

**uv.lock**: updated to pin new Moto commit with the fixes above

**.github/workflows/ci.yml**:
- `compatibility-tests` replaced with a 3-way matrix job (shards a-g, h-r, s-z)
  — each shard runs in parallel with its own server instance
- Workers increased: `-n4` → `-n8` (tests are I/O-bound HTTP calls)
- Removed `--reruns=2 --reruns-delay=1` (the root cause is now fixed; reruns
  were masking 500s, not fixing flaky tests)
- Timeout reduced: 20m → 12m per shard (expected: ~4-6min each)

**tests/compatibility/test_s3_compat.py**: trimmed to 92 tests (classes 1-25).
With `--dist=loadfile`, all 188 tests were bottlenecked to one worker.

**tests/compatibility/test_s3_ext_compat.py**: new file with remaining 96 tests
(classes 26+). Now two workers can process S3 tests in parallel.

### Expected speedup

- 3x from job parallelism
- ~2x from removing retry backoff (no more 500s)
- Combined: CI compat wall-clock time should drop from ~20min to ~4-6min per shard
