## Key decisions

PR #161 splits the single 15-minute app-integration-tests job into 14 parallel
matrix jobs. This exposed two pre-existing bugs that were previously hidden by
the combined job timing out before reaching those tests.

**Fix 1 — StepFunctions ExecutionHistory race (provider.py, asl.py)**

`_stop_execution` tried to record `ExecutionAborted` in the execution history, but
`_execution_histories[exec_arn]` was only populated *after* the background thread
finished. On slow CI machines the thread hadn't finished by the time `stop_execution`
was called, so `history = _execution_histories.get(exec_arn)` returned None and the
abort event was never recorded. `test_stop_execution_records_abort` would then assert
`'ExecutionAborted' in []` and fail.

Fix: pre-create `ExecutionHistory(exec_arn)` in `_start_execution` and register it in
`_execution_histories` *before* starting the thread. Pass the pre-created history into
`ASLExecutor` via a new optional `history` parameter so it reuses the same object
instead of creating a fresh one.

**Fix 2 — SQS FIFO dedup race (tests/apps/message_queue/conftest.py)**

The dedup tests called `broker.consume(fifo_queue, batch_size=10, max_messages=2, wait_time=5)`
expecting exactly 1 message back. The `fifo_queue` fixture used `visibility_timeout=5`.
After the first receive the message entered inflight with a 5s visibility window. The
consume loop immediately issued a second receive with `wait_time=5`. Within those 5s the
visibility expired and the background requeue worker put the message back, so the second
receive picked up the same message again — returning 2 instead of 1.

Fix: raise `visibility_timeout` in the `fifo_queue` fixture from 5s to 30s. Since
`wait_time=5 < 30s`, the message cannot reappear during a single consume call.
